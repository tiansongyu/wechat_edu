# 产品迭代与信任安全规则

> 文档状态：第 3 轮实施基线
> 更新日期：2026-07-16
> 适用范围：微信小程序、NestJS API、异步 Worker、Vue 管理后台、PostgreSQL 数据模型
> 原则：数据库是真相源；客户端不推导权限；先保证评价资格可信，再扩展互动和算法能力。

## 1. 系统当前用户旅程

### 1.1 家长

1. 微信登录，默认进入家长版，可完善常用地区和地址。
2. 发布家教需求，提交后进入后台审核。
3. 审核通过后接收老师报名，可接受或填写原因拒绝。
4. 接受报名后生成预约；家长可查看合作记录、联系老师、取消、发起争议或确认完成。
5. 家长也可以浏览老师发布的求带信息并建立平台会话。

### 1.2 老师

1. 从家长版明确切换到老师版。
2. 完善学校、学历、科目等资料并上传认证材料，等待后台审核。
3. 审核通过后可申请家教需求，或发布老师求带信息。
4. 报名被接受后建立预约和会话；老师可确认预约、取消或发起争议。
5. 双方确认完成后可互相评价；教师口碑只使用真实预约评价，旧静态 `score` 不再作为信誉信息展示。

### 1.3 管理员

1. 在独立 Web 后台登录。
2. 审核教师资料和家教发布。
3. 查看并处理报名、预约和争议状态。
4. 停用或恢复普通账号，查看审计日志。
5. 当前没有独立的评价审核、举报案件、拉黑和申诉工作台。

## 2. 对抗审查发现

### 2.1 评价资格不完整

- 当前预约只有家长侧“确认完成”语义，无法证明双方均认可履约完成。
- `TEACHER_OFFER` 目前主要形成会话，未必形成预约，因此不能直接授予评价资格。
- 仅聊天、收藏、申请、被拒绝或取消都不能代表真实履约。
- 如果只相信客户端传入的被评价人或角色，双角色账号可用错误身份评价。

### 2.2 重复评价和刷分

- 没有数据库唯一约束时，重试、连点和并发请求可能产生多条评价。
- 同一对用户反复制造短预约可能抬高或压低聚合分。
- 少量评价直接展示平均分会放大单条极端评价，并严重伤害新用户。

### 2.3 匿名和隐私

- 一对一合作中，被评价方通常可以推断评价人，产品不能承诺“绝对匿名”。
- 评价文本可能泄露手机号、微信号、住址、学校班级、儿童姓名或具体上课时间。
- 普通业务接口若直接返回完整教师资料，可能泄露真实姓名、审核意见或认证材料信息。

### 2.4 恶意差评与争议

- 履约争议中的评分不能作为稳定信誉依据。
- 低分不等于违规，不能因为被评价方不同意就自动删除。
- 管理员若能直接改写评价原文，会破坏证据完整性和用户信任。

### 2.5 角色切换误操作

- 双角色账号可能在详情页为了申请或联系而切换角色，但没有意识到全局身份已改变。
- 角色切换和实际业务命令若合并成一步，容易产生不可逆误操作。
- 历史评价身份不能随着用户之后切换角色而变化。

### 2.6 通知、空状态和信任安全

- 关键状态变化不能因为用户关闭普通提醒而完全不落站内通知。
- 加载失败、筛选无结果、首次使用和确实没有数据必须使用不同状态，不能统一显示“暂无”。
- 现有会话、账号停用和争议处理缺少完整的举报、拉黑、案件和申诉闭环。
- 当前静态教师分数不是履约计算结果，不应继续作为信誉指标展示。

## 3. 已选择的 MVP 需求与理由

| 已选择需求 | 产品规则 | 选择理由 |
| --- | --- | --- |
| 双方完成确认后互评 | 家长和老师都确认完成后，预约才进入 `COMPLETED` 并获得评价资格 | 比单方完成更能证明双方实际参与，且不依赖支付或签到系统 |
| 每人每预约一次 | 数据库唯一约束为 `appointmentId + reviewerId` | 抵御连点、请求重试和重复评价，规则容易解释 |
| 立即公开、角色匿名 | 评价创建成功后立即公开；普通用户只看到“本次合作家长/老师” | MVP 反馈及时，避免暴露昵称、头像和账号 ID |
| 少于 3 条不展示聚合星级 | 单条评价仍可展示；有效公开评价不足 3 条时 `displayAverage=null` | 避免一两条极端评价形成误导性总分 |
| 争议自动排除 | `DISPUTED` 预约不能新建评价；已存在评价停止公开计分，保留原始记录 | 不让未解决争议污染信誉，同时保留审计证据 |
| 后台可隐藏但不可改原文 | 管理员只能隐藏或恢复，必须填写原因并写审计日志 | 允许处理隐私和辱骂内容，但不允许平台篡改用户表达 |
| 评价内容最小化 | 1–5 星、最多 5 个结构化标签、可选短评；基础联系方式和隐私信息拦截 | 满足决策需要，同时降低人肉搜索和站外骚扰风险 |
| 角色由服务端推导 | reviewer、reviewee 和双方角色均从预约关系推导 | 防止客户端伪造被评价对象或错用家长/老师身份 |
| 明确角色切换 | 切换和业务操作分成两步，切换后必须重新确认操作 | 降低双角色账号误申请、误评价和误取消的概率 |
| 举报先做结构化案件 | 举报绑定目标类型、目标 ID、分类、提交人和处理状态 | 为后续拉黑、处罚和申诉提供可审计基础 |

## 4. 明确暂缓或拒绝的需求

### 4.1 本阶段暂缓

- **盲评窗口**：MVP 选择立即公开；待观察报复性评分比例后再决定是否引入双方提交后公开或定时公开。
- **AI 风控和 AI 自动审核**：不让模型自动判定评价真假、自动删评或永久封号。前期只做确定性校验、频率限制和人工审核。
- **公开评论区**：不做评价回复、楼中楼、点赞、点踩或围观讨论。评价是履约反馈，不是社区内容产品。
- **评价进入搜索排序**：在样本量、申诉和反刷机制稳定前，星级不参与职位或老师排序。
- **每节课评价**：当前没有课次、签到、支付和结算模型，只评价整个预约合作。
- **复杂加权信誉算法**：MVP 只提供有效评价数量、分布和简单平均；算法升级必须带版本号并可重算。
- **自动公开家长主页**：家长评价仅在必要业务场景提供汇总，不建立可搜索的公开个人评价主页。
- **完整聊天内容自动扫描**：本轮只做评价内容的确定性隐私拦截；消息内容审核另行评估合规和成本。

### 4.2 明确拒绝

- 允许仅聊过天、收藏过、报名被拒绝或取消的用户评价。
- 对平台和管理员也完全匿名的评价。
- 管理员直接修改用户星级或评价原文。
- 低于三星自动下架、自动封号或自动判定用户违规。
- 公开真实姓名、认证材料、详细地址、儿童信息或精确上课时间来证明评价真实。
- 删除评价后释放资格，让用户反复重新评价。
- 角色切换成功后自动继续提交评价、申请、取消等业务命令。
- 拉黑后让既有预约、争议入口和关键通知一并消失。

## 5. 五轮实际落地范围

### 第 1 轮：可信互评后端

目标是先形成可用、可审计、不会重复的服务端信任闭环。

范围：

- 为预约增加家长、老师双方完成确认时间；只有双方确认后才进入 `COMPLETED`。
- 新增评价数据模型、数据库唯一约束、评价状态和必要索引。
- 新增创建评价、查看账号评价和评价资格接口。
- 评价人、被评价人和角色全部由预约关系推导。
- 支持 1–5 星、角色标签白名单、最多 5 个标签、最多 500 字短评。
- 1–2 星评价要求填写不少于 10 个字的说明，并拦截常见联系方式和隐私信息。
- 公开输出不包含评价人 ID、昵称或头像；有效评价少于 3 条时不展示总星级。
- 争议预约自动排除；创建评价写 Outbox 和审计日志，并支持请求幂等。

#### 第 1 轮验收标准

- 任一方只确认一次时，预约仍不可评价；双方确认后才允许互评。
- 非预约参与者、错误活动角色、未完成、取消和争议预约创建评价均被拒绝。
- 同一用户对同一预约最多存在一条评价；同幂等键同内容返回原结果，同键不同内容返回 409。
- 公开响应中递归检查不到评价人 ID、昵称、头像、真实姓名和联系方式。
- 第 1、2 条有效评价时 `displayAverage` 为 `null`；第 3 条后才展示聚合星级。

### 第 2 轮：小程序评价与履约体验

范围：

- 新增评价页面，支持可点击星级、角色化标签、文字说明、字数和低分提示。
- 合作预约卡展示“待我确认、等待对方、双方已完成、待评价、已评价”等真实状态。
- 个人中心和需求详情展示基于真实评价的数量、等级和评价明细。
- 停止把静态教师 `score` 称为信誉分；评价不足时展示“评价积累中”。
- 加载、空结果、提交失败和无评价使用不同界面，不把异常伪装成空数据。

#### 第 2 轮验收标准

- 家长和老师都能从对应预约进入评价页，角色错误时只提示切换，不自动提交。
- 重复点击提交不会产生重复评价；服务端错误能保留用户已填内容。
- 1–5 星、标签、低分说明和 500 字限制与服务端规则一致。
- 第 3 条有效评价前不渲染误导性的平均星级。

### 第 3 轮：友好角色切换、平台信息与视觉统一

范围：

- 角色切换使用统一确认面板，解释切换后可以做什么，并在主要页面持续展示当前身份。
- 切换只刷新角色上下文，不自动继续申请、取消、评价等业务命令。
- 首页增加数据库驱动的平台概览、信任说明和新手引导，不展示伪造统计数字。
- 统一主要按钮触控高度、正文和辅助文字字号、卡片间距、圆角及可爱但克制的插画符号。
- 将“数据库同步”等工程文案改成用户能理解的业务语言。

#### 第 3 轮验收标准

- 首页、详情页和个人中心使用相同切换规则，取消确认后不发 API 请求。
- 平台统计来自服务端；无数据或接口失败时不显示虚假数字。
- 关键按钮和切换控件达到微信小程序可舒适触控的尺寸，长昵称和长标题不破版。
- 网络失败时不同时显示“暂无数据”。

### 第 4 轮：举报与后台评价治理

范围：

- 支持对评价提交结构化举报，分类包含隐私泄露、辱骂骚扰、虚假内容、广告导流和其他。
- 管理后台增加评价与举报工作台，支持筛选、分页、查看上下文、隐藏和恢复。
- 隐藏和恢复必须填写原因并写审计日志；管理员不能修改用户星级、标签或原文。
- 已隐藏、已移除或争议中的评价不进入普通列表和星级聚合。
- 普通用户只能查看自己的举报结果；管理员接口保持独立权限边界。

#### 第 4 轮验收标准

- 非法目标、重复未结举报和越权读取被稳定拒绝或聚合。
- 管理员隐藏评价但未填写原因返回 400；恢复后重新公开并计入聚合。
- 管理后台接口的加载、筛选空结果、失败和冲突状态均有明确反馈。
- 任何管理接口都不能改写原始评价内容或星级。

### 第 5 轮：对抗回归、地图可用性与发布硬化

范围：

- 修复地图降级样式、永久拒绝定位后的 `openSetting` 恢复入口和自定义 Tab 状态同步。
- 修复消息角标和时间重叠、过小文字和按钮、长表单反馈等跨设备问题。
- 为评价、举报、角色切换、履约、会话、审核和地址选择补充静态与 Docker 端到端回归。
- 验证数据库迁移、Worker 通知幂等、容器重启持久化和样例数据最小化。
- 本地测试临时使用 `127.0.0.1:4000`；真机测试提交使用服务器 ngrok HTTPS 地址。
- 保留评价算法版本和确定性规则，为以后同一合作对象影响上限、盲评和风控灰度预留空间。

#### 第 5 轮验收标准

- 五轮新增接口和原有核心接口全部通过自动化回归，Docker 服务保持健康。
- 本地开发、服务器体验版和真实微信登录三种配置不会被混淆提交。
- Docker 重启后评价、双方确认、隐藏状态、举报和账号资料保持一致。
- 最终提交的前端 API 地址是服务器配置，Git 工作区干净，且没有执行 push。

## 6. 核心产品规则

### 6.1 评价

1. 评价资格来自数据库中的预约参与关系，不接受客户端指定被评价人。
2. 双方完成确认是互评的必要条件。
3. 每人每预约仅能评价一次，记录不可物理覆盖。
4. 评价立即公开，作者仅显示角色匿名标签；平台和授权管理员保留追责能力。
5. 争议中的预约没有评价资格；已产生评价自动停止公开计分。
6. 用户可举报自己的对方评价，但低分本身不是违规理由。
7. 管理员只能隐藏、恢复或维持原状，不可修改原文、标签和星级。

### 6.2 星级

1. 星级范围为整数 1–5。
2. 聚合只统计 `PUBLISHED`、未隐藏、未争议排除且预约仍为 `COMPLETED` 的评价。
3. 有效评价少于 3 条时返回数量和分布，但不展示平均星级。
4. MVP 使用简单平均并返回 `algorithmVersion`，便于后续升级和重算。
5. 认证状态、评价星级和平台处罚是三种不同信息，不合并成单一“信誉分”。

### 6.3 举报与后台处置

1. 举报需要目标类型、目标 ID、分类和必要说明。
2. 举报内容仅举报人和授权管理员可见。
3. 隐藏评价必须关联处置原因；恢复同样写审计日志。
4. 管理员不得代表用户创建评价，不得改写用户内容。
5. 低分、批评和被评价方不同意都不能单独作为隐藏依据。
6. 严重安全问题走安全案件，不依赖评价区公开争论。

### 6.4 角色切换

1. 页面持续显示当前是家长版还是老师版。
2. 切换角色需要明确确认，服务端重新签发对应活动角色令牌。
3. 角色切换和申请、接受、取消、评价等命令不得合并执行。
4. 评价角色取自预约关系，并在评价创建时固化。
5. 通知需要另一角色时，只能引导切换，不能静默切换。

## 7. 测试矩阵

| 领域 | 场景 | 预期结果 |
| --- | --- | --- |
| 完成确认 | 仅家长确认完成 | 预约保持等待老师确认，不可评价 |
| 完成确认 | 仅老师确认完成 | 预约保持等待家长确认，不可评价 |
| 完成确认 | 双方先后确认 | 第二次确认后原子进入 `COMPLETED`，双方获得评价资格 |
| 角色权限 | 家长用老师版令牌评价家长 | 403，数据库无评价 |
| 角色权限 | 非预约参与人评价 | 404 或 403，不泄露预约存在性 |
| 资格 | PENDING、CONFIRMED、CANCELLED 预约评价 | 拒绝并返回稳定原因 |
| 资格 | DISPUTED 预约评价 | 拒绝；已有评价停止公开计分 |
| 重复提交 | 同一用户同一预约再次评价 | 409 或返回同一幂等结果，不新增记录 |
| 并发 | 50 个相同评价请求并发 | 最终仅一条评价和一条有效创建事件 |
| 幂等 | 同 key、相同规范化内容 | 返回第一次结果 |
| 幂等 | 同 key、不同内容 | 409，不覆盖第一次评价 |
| 内容 | 1–2 星且少于 10 字 | 400 |
| 内容 | 标签重复或超过 5 个 | 去重；超限或非法标签返回 400 |
| 内容 | 包含手机号、邮箱、微信号、身份证或 URL | 400，不写评价 |
| 匿名 | 普通用户读取评价列表 | 只见角色匿名标签，无 reviewer ID、昵称、头像 |
| 聚合 | 有效评价数量为 1 或 2 | `displayAverage=null`，count 和 distribution 正确 |
| 聚合 | 第 3 条有效评价创建 | 展示平均星级，分布与明细一致 |
| 聚合 | 一条评价被隐藏或进入争议 | 平均值和分布立即排除该评价 |
| 后台审核 | 管理员隐藏评价但未填写原因 | 400 |
| 后台审核 | 管理员隐藏后尝试改原文 | 接口不提供该能力或返回 403 |
| 后台审核 | 管理员恢复评价 | 重新公开并重新计入聚合，写审计日志 |
| 举报 | 重复举报同一评价 | 聚合到现有未结案件，不产生无限工单 |
| 拉黑 | 拉黑后创建新会话或发普通消息 | 403 |
| 拉黑 | 拉黑后处理既有预约争议 | 允许结构化处理并收到关键通知 |
| 角色切换 | 在评价按钮处切换角色 | 只完成切换和刷新，不自动提交评价 |
| 通知 | Outbox 或 Worker 重试 | 同一 `sourceEventId` 仅一条通知 |
| 空状态 | API 加载失败 | 显示错误和重试，不显示“暂无评价” |
| 隐私投影 | 深度检查普通 API 响应 | 不含真实姓名、审核意见、证件地址、精确隐私字段 |
| 持久化 | Docker 重启 | 评价、双方确认、隐藏状态和举报均保留 |
| 微信调试 | 家长、老师、双角色完整流程 | 页面可操作、状态同步、无控制台错误和未处理 Promise |
| 管理后台 | 评价列表、隐藏、恢复、筛选空结果 | 状态准确；加载失败与空结果明确区分 |

## 8. 五轮整体发布门槛

五轮代码只有在以下条件全部满足后才建议开放给真实用户：

- 数据库迁移可重复部署，旧预约不会被错误授予评价资格。
- 所有创建、完成确认、争议和隐藏操作都有越权及并发测试。
- 公开评价响应通过递归敏感字段检查。
- 评价聚合可以从明细全量重算，结果与在线查询一致。
- 管理员无法修改评价原文。
- 小程序完成家长和老师双向流程验证。
- 微信开发者工具中无业务报错、无失效路由、无重复请求导致的重复评价。
- Docker Compose 全部服务健康，API、Worker、管理端和数据库重启后状态一致。
